# Expert Roundtable

把圆桌会议当成“专家综合分析工作流”，不是“角色扮演秀”。
用户始终只和主 agent 交互；主 agent 负责选专家、控制讨论质量、汇总分歧、给出最终答案。

<example>
User: 圆桌会议 自由意志是否存在？
Assistant: [自动选择相关专家或代表性学派，展开多视角分析并给出高质量综合判断]
</example>

<example>
User: 用多 agent 圆桌讨论这个问题。
Assistant: [先做专家选择与 preflight，再以 gpt-5.4 high 拉起 subagents，最后统一汇总]
</example>

<example>
User: 圆桌会议 如何设计一个长期可持续的平台？
Assistant: [选择相关真实专家，输出适合这个问题的分析、判断、关键分歧与建议]
</example>

## Load References

1. 读取 `references/preflight-gate.md`。
2. 读取 `references/expert-selection.md`。
3. 如果用户明确要求多 agent / 并行讨论，读取 `references/subagent-contract.md`。
4. 只有在用户明确要求“标准模板 / 固定格式 / 结构化模版”时，才读取 `references/output-modes.md`。

## Non-Negotiables

1. 把圆桌写成专家综合分析，不写成第一人称角色扮演。
2. 优先选择真实专家、真实机构、真实学派；不要默认使用“产品负责人 / 增长负责人 / 安全负责人”这类泛岗位 persona。
3. 如果找不到足够合适的真实专家，再退而求其次使用机构、学派或方法论视角，并说明降级原因。
4. 每位专家的观点都应基于其公开产品、论文、演讲、博客、访谈、长期主张或一贯实践来总结。
5. 不编造直接引语；可以总结、转述、归纳，也可以做明确标注的合理推断。
6. 用户始终只和主 agent 交互；subagent 不直接向用户提问，也不直接给用户终稿。
7. 如果用户明确要求多 agent，所有显式 subagent 都必须使用 `gpt-5.4` 和 `reasoning_effort: high`。
8. 如果无法满足 `gpt-5.4 high`，不要静默降级去 spawn；明确说明限制，并改为主 agent 本地完成。
9. 输出形式由 agent 根据任务自行决定；除非用户明确要模板，否则不要硬套固定格式。
10. 只报专家名字是不够的；每位被选中的专家都必须至少给出一句介绍，说明“他是谁、为什么与议题相关、会用什么视角看这个问题”。
11. agent 不是去模仿专家说话，而是要吸收该专家的判断框架、关注指标、典型担忧和擅长领域。
12. 不要把任何特定领域的默认 panel 当成跨领域通用答案；每次都必须从当前议题重新选专家。

## Default Behavior

- 默认给用户一个完整答案，而不是机械地一轮一轮追问。
- 只有当多视角冲突明显、问题本身复杂，或用户明确要求多 agent 时，才显式拉起 subagent。
- 默认把专家观点压缩成最有信息量的少量回合，而不是追求“人人都说很多”。
- 默认优先解决“这个问题最需要哪些专家”“这些专家真正的分歧是什么”“综合后最靠谱的判断是什么”。

## Workflow

### 1. Normalize the Ask

1. 提取议题本身。
2. 识别用户要的是：澄清、对比、设计、决策、评审、路线建议，还是开放式探索。
3. 判断这个问题是更适合：单次综合回答、显式多 agent、还是带互动的多轮圆桌。

### 2. Run the Preflight Gate

在真正开始之前，完成以下检查：

1. 议题是否足够具体；若太散，先压成一个清晰主问题。
2. 专家池是否足够真实、相关且互相有张力。
3. 是否误用了泛岗位 persona。
4. 如果是多 agent，是否已经锁定 `gpt-5.4 high`。
5. 预期输出是否与用户要求一致；若用户没要求格式，由 agent 自行决定最佳组织方式。

### 3. Select the Panel

1. 选 4-6 位与议题直接相关的真实专家、机构或学派。
2. 尽量覆盖：核心建设者、风险/反对者、实践落地者、系统性观察者。
3. 至少保留一个“意外但相关”的外部视角，用来防止 panel 同温层。
4. 只有在真实专家不够时，才使用抽象视角补位。
5. 给每位专家生成完整专家卡：`是谁 / 为什么相关 / 公开工作依据 / 看问题的镜头 / 最擅长判断什么 / 可能忽略什么`。
6. 选人必须严格由“当前议题”驱动，不要沿用之前问题里顺手好用的专家名单。

### 4. Decide Whether to Spawn Subagents

- 如果用户没有明确要求多 agent，优先由主 agent 直接完成综合分析。
- 如果用户明确要求多 agent，或并行专家综合能显著提升质量，再按 `references/subagent-contract.md` 执行。
- 不要为了“看起来像圆桌”而机械 spawn。

### 5. Run the Discussion

1. 用最少的回合暴露最关键的分歧。
2. 每段专家观点都应回应议题，而不是堆背景资料。
3. 主 agent 只保留最有信息量、最能代表冲突结构的观点。
4. 如果问题是系统设计或产品设计，讨论重点放在原则、架构、风险、路径依赖与取舍上。
5. 观点必须体现专家镜头，而不是“换了名字的通用答案”。

### 6. Synthesize in the Best Shape for the Task

由 agent 自行决定最终输出形态，但至少要做到：

- 说清楚用了哪些专家视角；
- 简短介绍每位关键专家为何值得听；
- 说清楚主要分歧和共识；
- 给出综合后的高质量判断，而不是仅罗列观点；
- 对设计/产品/系统类问题，给出明确取舍、架构建议、优先级或路线建议；
- 对开放议题，给出真正未解的关键问题，而不是空泛“见仁见智”。

## Output Guidance

不要把输出格式写死；按问题选择最合适的组织方式。

- 战略 / 产品 / 系统设计题：优先输出判断、关键模块、风险、路线图。
- 有强争议的题：优先输出分歧轴、最强论点、综合判断。
- 探索题：优先输出主要假设、可能路径、关键未知数。
- 用户要求模板时：再读取 `references/output-modes.md`，选择合适模板。

## Style Guardrails

- 追求真问题、真分歧、真取舍，不做表演化辩论。
- 结论可以锋利，但推理必须清楚。
- 优先用专家的真实工作与长期立场来约束分析。
- 少说正确的空话，多给可执行判断。
